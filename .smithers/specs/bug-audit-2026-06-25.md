### 1. [electric-openapi / security] Electric `docs` shape forwards the entire `_smithers_docs` table unscoped to any run:read principal (fail-OPEN)
**File:** `packages/electric-proxy/src/smithersElectricShapeCatalog.ts:64-69`

**What:** The `docs` shape (table `_smithers_docs`) is defined with NO `runIdColumn`, `workspaceIdColumn`, `userPrivateColumn`, and NO `whereTemplate`. In `validateWhere` (createSmithersElectricProxy.ts:297-328) the fail-closed guard at line 308 only fires when one of those three scoping columns is set; for the `docs` shape it is skipped, so an empty client `where` causes `validateWhere` to return `null`, `sanitizeQuery` deletes the `where` param, and the proxy forwards an UNFILTERED read of the whole `_smithers_docs` table (tickets, plans, specs, proposals). The schema (packages/db/src/sql-message-storage.js:357-365) confirms `_smithers_docs` has no `run_id`/`workspace_id` column, so it cannot be scoped by the existing mechanism at all. The cloud bin (bin/smithers-electric-proxy.ts) grants only `scopes:['run:read']` + `grantedRunIds`, so any authenticated token that can read even one run can stream every doc in the store — across runs and, in a multi-tenant Electric/Postgres deployment, across tenants. This is the exact fail-open the rest of the module is designed to prevent.

**Why it is a bug:** The module's stated contract (SmithersElectricAuthContext doc comment lines 23-30 and the catalog comment) is fail-CLOSED: a scoped shape with no concrete grant is rejected, never forwarded unscoped. The `docs` shape silently violates that because the guard keys off scoping columns that the shape lacks, so it is treated as a globally-readable table for any run:read principal. There is no code path anywhere that rejects a shape which has no row-level scoping, and no test covers the `docs` shape.

**Trigger:** GET /v1/shape?table=_smithers_docs with any token that authenticates with scopes:['run:read'] (e.g. one granted only run-1). Response: 200, streaming every row of _smithers_docs with no run/tenant filter.

**Fix:** Treat a shape with no scoping column (and no whereTemplate) the same as the fail-closed branch: reject unless `auth.unscoped === true`. Concretely, in validateWhere change the line-308 condition to also fail closed when the shape has no scoping column at all, or give the `docs` shape a real scoping column/template (and drop it from the catalog until `_smithers_docs` carries a run_id/workspace_id).


### 2. [react-reconciler / data-loss] diffSnapshots duplicates a node when it reparents into a re-added subtree
**File:** `packages/devtools/src/diffSnapshots.js:180-203`

**What:** When a node moves to a new parent and that new parent lives inside a subtree that is itself being removed-and-re-added (because an ancestor moved or changed shape), diffSnapshots emits BOTH the ancestor's addNode (whose cloned subtree already contains the moved node at its final position) AND a separate standalone addNode for the moved node. applyDelta executes both, inserting a second copy, so the patched tree contains a duplicated node id and the round-trip diffSnapshots(a,b) -> applyDelta(a,delta) no longer equals b.

**Why it is a bug:** topLevelAddIds (lines 180-183) suppresses a standalone addNode only when the node's DIRECT parent (to.get(id).parentId) is in addSet. But a node is also implicitly re-added whenever ANY ancestor is in addSet, because addNode at lines 195-202 clones the entire b-subtree (cloneValue(entry.node)) including descendants. If the node's direct parent is an UNCHANGED node (same type/name/depth and same parent/index, so not in addSet) that merely happens to sit inside a re-added ancestor subtree, the filter does not catch it. applyDelta then both clones the node in via the ancestor's addNode and splices it again via the standalone addNode, producing a duplicate. CONSEQUENCE: diffSnapshots is the server-side delta producer in packages/server/src/gatewayRoutes/streamDevTools.js (devtools.diffSnapshots) and the deltas are consumed by applyDelta in packages/server/src/gatewayUi/defaultOperatorUi.js, packages/pi-plugin/src/runtime/DevToolsStore.ts, and apps/cli/src/tree.js (smithers tree). applyDelta does not detect duplicate ids, so the corrupted tree silently streams to every live DevTools/UI client and the CLI tree view whenever a workflow node reparents across a commit.

**Trigger:** FROM root 1 with children [5, 2->[3]] (5 at index 0, 2 at index 1). TO root 1 with children [2->[3->[5]]]. Node 2's index changes 1->0 so 2 is removed+re-added with a full clone (clone contains 5 under 3). Node 3 keeps the same shape/position relative to parent 2 so 3 is NOT in addSet. Node 5 reparents 1->3 so it is in removeSet+addSet; its direct parent 3 is not in addSet so its standalone addNode is emitted. delta.ops = [removeNode 5, removeNode 2, addNode(2 under 1 with full clone), addNode(5 under 3)]. applyDelta yields node 5 twice under node 3. Fuzzing 200k random reparenting tree-pairs reproduced this in ~2.6% of cases, always producing duplicate (never lost) nodes.

**Fix:** Filter a node out of the standalone addNode set when ANY ancestor (not just the direct parent) is in addSet: walk the parentId chain in `to` upward from each add candidate and exclude it if any ancestor id is in addSet (equivalently, only emit addNode for nodes whose nearest re-added/moved ancestor is the node itself). Add a regression test for cross-parent moves into a re-added subtree.


## HIGH (21)

### 3. [app-observability / security] API-key redaction misses hyphenated OpenAI/Anthropic key formats — secrets leak into trace artifacts and OTLP logs
**File:** `apps/observability/src/_traceRedaction.js:6-33`

**What:** The `api-key` redaction rule is `\b(?:sk|pk)_[A-Za-z0-9_-]{8,}\b`, which requires an underscore after `sk`/`pk` (Stripe-style). Real OpenAI keys (`sk-...`, `sk-proj-...`) and Anthropic keys (`sk-ant-api03-...`) use a HYPHEN after `sk`, so they never match this rule. The only other rule that could catch a bare key is `secret-ish` (`\b(?:api[_-]?key|token|secret|password)=...`), which only fires when the key is immediately preceded by `key=`/`token=`/`secret=`/`password=`. A bare key in agent output (an agent echoing `$OPENAI_API_KEY`, a logged curl command `curl -H 'x-api-key: sk-ant-...'`, an error message containing the key) is not redacted by any rule. Additionally `secret-ish` requires a `\b` word boundary before `api`, so an env-style dump like `ANTHROPIC_API_KEY=sk-ant-...` matches NOTHING (the `_` before `API` is a word char, so no boundary) — the key leaks via both rules. `redactValue` is the sole scrubbing layer in `AgentTraceCollector.push`/`observeProviderSessionRow` before payloads/raw rows are persisted to the canonical NDJSON trace artifact and exported to OTLP, so unredacted keys are durably written to disk and shipped to the collector.

**Why it is a bug:** Verified empirically: `redactValue('OpenAI key sk-proj-abc12345678')`, `redactValue('curl -H x-api-key: sk-ant-api03-AbCdEf123456')`, and `redactValue('ANTHROPIC_API_KEY=sk-ant-xyz9876543')` all return `{applied:false}` and leave the key intact. Only `sk_`/`pk_` underscore forms (Stripe) or `token=`/`key=`-prefixed values are scrubbed. The hyphenated forms are the de-facto standard for the two most common providers Smithers drives (Claude/OpenAI).

**Trigger:** An agent (e.g. ClaudeCodeAgent or CodexAgent) emits a line containing a real API key, e.g. stdout `Using ANTHROPIC_API_KEY=sk-ant-api03-XXXX` or a tool result echoing `Authorization: x-api-key sk-proj-XXXX`. The trace collector calls redactValue on the payload/raw, the key survives, and it is written to the NDJSON artifact and sent to OTLP.

**Fix:** Broaden the `api-key` pattern to cover hyphenated and provider-namespaced keys, e.g. `/\b(?:sk|pk)[-_](?:[A-Za-z0-9]+[-_])*[A-Za-z0-9_-]{8,}\b/g` (covers `sk-`, `sk_`, `sk-ant-`, `sk-proj-`), and make the `secret-ish` left side tolerant of an underscore-joined prefix (drop the leading `\b` or use `(?:^|[^A-Za-z0-9])` so `ANTHROPIC_API_KEY=` matches). Add explicit redaction tests for `sk-ant-...`, `sk-proj-...`, bare `sk-...`, and `NAME_API_KEY=...`.


### 4. [app-review / data-loss] Over-cap usage is silently dropped from both spend tally and the usage_events audit log
**File:** `apps/review/src/server/proxy/recordUsage.ts:31-39`

**What:** recordUsage increments the session spend with a conditional UPDATE `... WHERE hash = ? AND spent_usd + ? <= spend_cap_usd`. When a call's cost would cross the cap, the UPDATE matches zero rows, the function returns {recorded:false}, and execution returns BEFORE the `INSERT INTO usage_events` runs. So that call's cost AND its token counts are written nowhere. handleAnthropic discards the return value (the whole metering IIFE is `.catch(() => undefined)` and `deps.waitUntil`-ed with no inspection), so there is no log, no metric, and no audit row. The proxy has already forwarded the request and streamed Anthropic's full response to the client (real billable spend at Anthropic), yet the service records nothing for it.

**Why it is a bug:** The pre-flight check in handleAnthropic (`spentUsd >= spendCapUsd → 402`) only blocks the NEXT request; it cannot un-spend a call already in flight. The conditional UPDATE then refuses to record the in-flight call's true cost because recording it would push spent_usd past the cap. The result is that operator dashboards, the `/metrics` review_spend_usd_total gauge, and `/api/admin/usage` systematically undercount real Anthropic spend for every session that crosses its cap. The existing test 'records concurrent session usage without exceeding the spend cap' (handleAnthropic.test.ts:203-237) hard-codes this lossy behavior: two 200-OK billable calls, but only one usage_events row and spent_usd == one call's cost.

**Trigger:** Seed a session with spend_cap_usd just above one call's cost, then fire two concurrent /anthropic/v1/messages requests (or run a real review whose cumulative cost crosses the cap mid-stream). Both return 200 with full streamed bodies, but exactly one usage_events row is written; the second call's cost and tokens are lost.

**Fix:** Always INSERT the usage_events row (it is the audit ledger of money actually spent at Anthropic) and increment spent_usd unconditionally with `spent_usd = spent_usd + ?` (clamp the cap separately for the gate). Treating the cap as a hard ledger filter conflates 'should we have allowed this' with 'how much did we actually spend'. If the goal is to mark over-cap usage, add a flag column rather than dropping the row.


### 5. [cli-dispatch / correctness] parseJsonInput(...,fail) does not abort the handler on malformed JSON; the error sentinel is used as the parsed value and the command performs its side effect with corrupt data
**File:** `apps/cli/src/index.js:5141`

**What:** In the `signal` handler, `const payload = parseJsonInput(c.options.data, "signal data", fail) ?? {}`. parseJsonInput (json-args.js:48) on a JSON.parse failure calls `fail(...)`, and `fail` is `(opts) => { commandExitOverride = opts.exitCode ?? 1; return c.error(opts); }`. Under `cli.serve`, incur's `c.error()` returns a plain truthy sentinel object `{ [Symbol(incur.sentinel)]:'error', code, message, exitCode }` (incur Cli.js:924) and does NOT throw. So `parseJsonInput` returns that sentinel; `?? {}` does not apply (the sentinel is truthy), and the handler keeps running: it calls `adapter.getRun(...)` and then `signalRun(adapter, runId, signalName, payload, ...)` with the error-sentinel object as the signal payload, then returns `c.ok({ status:"signalled" })`. The waiting run is signalled with a garbage payload and a success envelope is printed to stdout. (commandExitOverride was set to 4 so the process still exits 4, but the destructive side effect already happened and stdout shows success.) The same misuse occurs at lines 4745 (`human answer` --value), 5408 (replay --input), 5842 (replay --input), and 6099 (fork --input): each assigns the parseJsonInput result and continues, feeding the error sentinel as the run input override / human-answer value to a DB write.

**Why it is a bug:** parseJsonInput's `fail` callback does not throw and incur's c.error returns (not throws) a truthy sentinel, so every caller that writes `const x = parseJsonInput(...)` instead of `return`-ing it on failure continues executing with the sentinel object in place of the parsed JSON. The signal/fork/replay code paths have no schema guard, so the sentinel is persisted/delivered verbatim. The fix used elsewhere (executeUpCommand at 1728-1761) wraps parseJsonArgument in try/catch and returns fail(...) — the parseJsonInput(...,fail) call sites skip that and just use the return value.

**Trigger:** Run `smithers signal <runId> myEvent --data '{not-json'` against a run waiting on an event: the run receives a corrupt signal payload (the incur error sentinel) and stdout prints a `status: "signalled"` success envelope (process still exits 4). Likewise `smithers fork wf.tsx -r <runId> --frame 0 --input '{bad'` forks a run whose inputOverrides is the sentinel object; `smithers human answer --value '{bad'` answers a schemaless human request with the sentinel object.

**Fix:** Make the call sites abort on failure, e.g. detect the sentinel/`undefined` and return it: `const payload = parseJsonInput(c.options.data, "signal data", fail); if (isErrorSentinel(payload)) return payload; const data = payload ?? {};`. Better, have parseJsonInput throw a SmithersError (or have callers wrap it in try/catch + `return fail(...)`) so the handler short-circuits before any side effect, mirroring executeUpCommand.


### 6. [cli-runlife / correctness] `smithers cancel` cannot stop a live run — never sets cancelRequestedAtMs, so the engine keeps running and clobbers the cancelled status
**File:** `apps/cli/src/index.js:5266-5295`

**What:** The `cancel` command marks in-progress/waiting-timer attempts cancelled and writes `updateRun(runId, { status: "cancelled", finishedAtMs })`, but it never calls `adapter.requestRunCancel(...)` (i.e. it never sets `cancel_requested_at_ms`). The only mechanism the running engine has to detect a durable cancellation is its cancelWatcher polling `cancel_requested_at_ms` (packages/engine/src/engine.js:1919 `if (run?.cancelRequestedAtMs) { ... controller.abort(); }`). A direct status flip is invisible to that watcher.

**Why it is a bug:** For a LIVE detached / `up --serve` run (the primary target of `cancel` — its CTA points at `ps`/`logs`, and the doc at apps/cli/docs/SKILL.md:370 says `smithers cancel <run-id>  # stop a run`), the engine process never observes the cancel: the heartbeat keeps running (heartbeatRun touches only heartbeat_at_ms, db/adapter.js:918), agents keep executing, and when the engine completes it writes `status: "finished"` (engine.js:5469-5476), fully overwriting the cancellation. So `cancel` neither stops the agents nor durably marks the run cancelled. The server's own cancel route proves the correct contract: for a fresh running run it calls `adapter.requestRunCancel(runId, nowMs())` + `abort.abort()` (packages/server/src/serve.js:287-291) and only does the bare status-flip for stale/non-fresh runs. The CLI omits the live-run branch entirely. `down --force` (index.js:5350-5370) shares the same bare-flip flaw for live runs.

**Trigger:** Start a detached or served run: `smithers up wf.tsx -d --run-id r1` (a long-running agent task). While it is actively running (fresh heartbeat), in another shell run `smithers cancel r1`. The DB momentarily shows status=cancelled, but the engine process keeps executing the agent and on completion rewrites status=finished; the agents were never halted.

**Fix:** In the cancel handler, branch on liveness like serve.js: if `run.status === "running" && isRunHeartbeatFresh(run)`, call `await adapter.requestRunCancel(c.args.runId, Date.now())` so the engine's cancelWatcher aborts the live run, and only do the direct attempt/run status flip for stale runs. Apply the same fix to `down --force`.


### 7. [cli-workflow-pack / error-handling] One malformed workflow file aborts ALL workflow discovery (poison-pill)
**File:** `apps/cli/src/workflows.js:320-326`

**What:** discoverWorkflows() calls workflowFromFile()->parseMetadata() for every .tsx in the workflows dir inside the loop with no per-file try/catch. parseMetadata throws SmithersError("INVALID_WORKFLOW_METADATA") for any file whose `// smithers-metadata-version:` comment is not exactly "1" (and any other parse failure propagates too). A single bad file therefore makes the whole discovery throw, so NONE of the user's other (valid) workflows are listed/runnable.

**Why it is a bug:** The metadata-version check is explicitly a forward-compat guard ('Unsupported workflow metadata version'). Its intent is to reject one unsupported file, but because the throw happens inside the discovery loop and discoverWorkflows() is unguarded, it instead takes down the entire workflow surface. Verified by repro: a dir with good.tsx (version 1) + future.tsx (version 2) makes discoverWorkflows() throw INVALID_WORKFLOW_METADATA and return nothing. discoverWorkflows is called unguarded by `workflow list` (index.js:2299), `workflow run`/`path`/`inspect` via resolveWorkflow (workflows.js:348), and the multi-run gateway/up --serve workspace registration (index.js:2189) — the per-workflow loadWorkflow there is wrapped in try/catch but discoverWorkflows itself is not, so the whole gateway crashes before registering anything.

**Trigger:** Author or hand-edit any `.smithers/workflows/<name>.tsx` with `// smithers-metadata-version: 2` (e.g. a workflow written against a newer smithers, or a corrupted header). Then `smithers workflow list` / `smithers gateway` fail entirely instead of skipping just that file.

**Fix:** Wrap the per-file workflowFromFile() call in discoverWorkflows in a try/catch that records/skips the offending file (e.g. push a warning to stderr and continue) so one bad file never hides the rest. Optionally surface skipped files in the result.


### 8. [control-plane / data-loss] Crash mid secret-ref migration silently loses ALL secret references on recovery
**File:** `packages/control-plane/src/index.js:183-224`

**What:** migrateSecretRefsProjectKey() runs RENAME -> CREATE -> INSERT...SELECT -> DROP as four separate auto-committed statements via sqlite.exec() with NO enclosing transaction. If the process crashes after the RENAME (or after CREATE but before/within the INSERT...SELECT), the database is left with the renamed _smithers_cp_secret_refs_legacy table holding the real data and no _smithers_cp_secret_refs. On restart, ensureControlPlaneTables() first runs CREATE TABLE IF NOT EXISTS _smithers_cp_secret_refs (which recreates an EMPTY table that already has project_key), then migrateSecretRefsProjectKey() sees the project_key column present and returns immediately. The legacy rows in _smithers_cp_secret_refs_legacy are never re-imported and are orphaned forever.

**Why it is a bug:** I reproduced it: created a file DB with a legacy NULL-project secret ref 'vault://critical-prod-secret', executed only the RENAME step (simulating a crash), then re-opened the DB and called ensureControlPlaneTables(). Result: _smithers_cp_secret_refs is empty (0 rows), _smithers_cp_secret_refs_legacy still holds the secret, and the migration guard short-circuits because project_key now exists on the freshly-created empty table. The org's entire secret-reference catalog is lost from the control plane's point of view. SQLite auto-commits each top-level statement, so partial DDL/DML progress survives a crash; the guard's reliance on column existence rather than a completed-migration marker makes recovery skip the un-migrated data.

**Trigger:** Open an older control-plane SQLite DB whose _smithers_cp_secret_refs lacks the project_key column; crash/kill the process after the ALTER TABLE RENAME executes but before DROP. Re-open the DB (any normal startup) -> all secret refs vanish.

**Fix:** Wrap the whole migration body in an explicit transaction (BEGIN IMMEDIATE; ... ; COMMIT) so it is atomic, AND make the migration self-healing: if _smithers_cp_secret_refs_legacy exists at start of ensureControlPlaneTables, re-run/finish the migration from it before the CREATE IF NOT EXISTS recreates an empty table. Use sqlite.transaction(...) (bun:sqlite) or a manual BEGIN/COMMIT/ROLLBACK around the statements.


### 9. [control-plane / security] Project literally named "__org__" collides with org-wide sentinel: org-wide secrets and usage limits are silently overwritten / cross-leaked
**File:** `packages/control-plane/src/index.js:342-344`

**What:** projectKey(projectId) returns the literal sentinel "__org__" for org-wide (projectId === null) rows, and this value is used as part of the primary key for both _smithers_cp_secret_refs (PK org_id, project_key, name) and _smithers_cp_usage_limits (PK org_id, project_key, metric, unit, period). But ID_RE (/^[A-Za-z0-9:_-]{1,128}$/) accepts "__org__" as a valid project_id and nothing reserves it. So a real project with projectId === "__org__" maps to the SAME project_key as org-wide scope.

**Why it is a bug:** Reproduced: created org-wide secret ref 'vault://ORG-WIDE' (projectId null) then a project-scoped secret for projectId '__org__' with the same name -> the org-wide row's DELETE-then-INSERT keys on (org_id,'__org__','tok'), so the org-wide secret is destroyed and only 'vault://PROJECT-SCOPED' remains. listSecretRefs({projectId:null}) (org-wide) now returns the project-scoped secret, and listSecretRefs({projectId:'__org__'}) returns the same row: a cross-scope leak AND data loss. Identical collision for usage limits: an org-wide limit of 100 is overwritten by a project '__org__' limit of 7. assertProjectExists does not stop this because the project '__org__' genuinely exists.

**Trigger:** store.createProject({orgId, projectId:'__org__', slug, name}); then store.putSecretRef({orgId, name:'tok', ref:'org-wide'}) followed by store.putSecretRef({orgId, projectId:'__org__', name:'tok', ref:'scoped'}). The org-wide secret ref is gone and the two scopes read each other's secrets.

**Fix:** Reject "__org__" (and ideally any value containing the reserved prefix) in optionalId/requiredId when validating project_id, or use a sentinel that cannot be a valid project_id (e.g. an empty string disallowed by ID_RE, or a value containing a character ID_RE forbids). Validate the sentinel is unreachable from user-supplied project ids.

### 12. [electric-openapi / correctness] Free-form object request bodies are silently stripped to `{}` by jsonSchemaToZod (data loss on tool calls)
**File:** `packages/openapi/src/jsonSchemaToZod.js:151-156`

**What:** `buildObject` only attaches a `.catchall(...)` when the schema has declared `properties` (`Object.keys(props).length > 0`). For a free-form body schema like `{ type: "object" }` or `{ type: "object", additionalProperties: true }` with no `properties`, the generated Zod schema is `z.object({})`, which STRIPS every key on parse. The AI SDK validates/parses tool args against this schema before calling `execute`, so the LLM-supplied request body `{foo:1,bar:'x'}` is parsed down to `{}` and `executeRequest` (tool-factory/_helpers.js:218-221) serializes `{}` as the HTTP body. The real payload never reaches the upstream API.

**Why it is a bug:** Verified empirically: `jsonSchemaToZod({type:'object'}).safeParse({foo:1,bar:'x'})` returns `{success:true,data:{}}`, and `jsonSchemaToZod({type:'object',additionalProperties:true})` does the same. Free-form/open object bodies are extremely common in real OpenAPI specs (webhook payloads, metadata blobs, generic `data` objects). The catchall guard incorrectly requires declared properties; an empty-property object with additionalProperties true/undefined should permit unknown keys.

**Trigger:** Generate a tool from any operation whose requestBody schema is `{ type: 'object' }` (no `properties`). Invoke it with a JSON body; the upstream API receives `{}`.

**Fix:** When `additionalProperties` is true/undefined, always add a catchall (or use `z.record(z.unknown())` when there are no declared properties), regardless of whether `props` is empty: drop the `Object.keys(props).length > 0` condition at line 153.
