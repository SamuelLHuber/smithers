Primer

What iron-proxy is

A default-deny egress firewall for untrusted workloads: CI jobs, sandboxed containers, and AI coding agents. Open-sourced from the Optimism stack by @mslipper. Single Go binary, Apache-2.0.

It sits in front of everything the workload sends out and answers one question per request: is this allowed, and what should it actually carry?

  • Default-deny allowlist. Every outbound request is blocked unless the host matches a domain glob or CIDR.
  • Boundary secret rewriting. The proxy holds the real secrets. The sandbox only ever sees placeholder tokens, swapped for real ones on the way out.
  • SSRF / metadata block. Even allowlisted hosts are denied if they resolve into 169.254.169.254 or loopback.
  • Per-request audit. Every request becomes structured JSON: host, action, status, which secret swapped, which rule matched.

Transform pipeline

An ordered chain of policy stages over each request and response:

  • allowlist · default-deny, with a warn-only mode
  • secrets · token → real value from env / file / AWS / 1Password
  • header-allowlist · strip headers before upstream
  • judge · ask an LLM to allow / defer a risky request
  • mcp-policy · default-deny tool allowlist for MCP servers
  • annotate / body-capture · enrich the audit log

DNS :53 · HTTP/HTTPS proxy :80/:443 (TLS via on-the-fly leaf certs) · CONNECT + SOCKS5 tunnel · OTEL export

The gap today

Untrusted agents, unrestricted egress

Both products run code an attacker can influence through prompt injection or a poisoned dependency. Right now that code can phone home freely and read secrets in the clear.

Plue · Smithers Cloud

  • Agents run in Freestyle microVMs. agent_dispatch.buildServiceSpec sets no firewall: open egress to any IP and port.
  • Secrets are injected as plaintext env (SMITHERS_AGENT_TOKEN, repo secrets). Visible to agent code via env, ps, memory.
  • A deny-default FirewallPolicy already exists, but only workflow sandboxes use it. Agent VMs do not.

Smithers · framework

  • CLI agents (claude-code, codex) spawn as subprocesses; SDK agents call the ai SDK. Both reach providers directly.
  • Provider keys live in process.env and flow into every spawned agent's environment.
  • No network isolation of agent subprocesses. A compromised tool can exfiltrate keys or open a reverse shell.

Same failure mode, two layers: the workload can talk to anywhere, and it can read the keys it talks with.

The shape you asked for

One seam. Two postures.

Smithers framework

A clean, pluggable integration point that lets any egress tool slot in trivially, and that never names iron-proxy.

Core defines a port. iron-proxy, mitmproxy, Squid, a corporate proxy, or nothing at all are interchangeable adapters. Open-source users bring their own or run wide-open.

Plue · Smithers Cloud

Registers iron-proxy as the default adapter. Every cloud agent gets default-deny egress, secret rewriting, and audit with zero opt-in.

The framework stays vendor-neutral. The hosted product makes a strong default choice on top of it.

Smithers core → defines → EgressProvider port ← implements ← iron-proxy adapter ← default in ← Plue Cloud

Ports & adapters. Dependencies point inward; core depends on nothing.

Why this is not a big lift

The seam is already a standard

Every proxy worth using already speaks one contract:

HTTP_PROXYHTTPS_PROXYNO_PROXYNODE_EXTRA_CA_CERTS

Set those four values plus a trusted CA, and a process routes all egress through the proxy, including TLS. That is the whole vendor-neutral surface. Smithers does not need an iron-proxy SDK; it needs to thread proxy env + a CA into every place an agent makes a request.

Iron-proxy's secret rewriting is the bonus on top: because the proxy terminates TLS, it can swap a placeholder for a real key. Smithers gets that for free by honoring the same contract.

Smithers core · no vendor names

The EgressProvider port

packages/driver/src/egress/EgressProvider.ts

// Smithers CORE. Knows nothing about iron-proxy or any vendor.
export interface EgressContext { runId: string; nodeId: string; agentKind: string; workspaceId?: string }

export interface EgressBinding {
  /** Env overlay merged into every agent process and subprocess. */
  env: Record<string, string>;          // HTTP_PROXY, HTTPS_PROXY, NO_PROXY, NODE_EXTRA_CA_CERTS
  caCertPem?: string;                      // CA to trust for TLS interception, if any
  secretBindings?: Record<string, string>;   // "sk-proxy-XXX" -> "anthropic"  (real value resolved at the boundary)
}

export interface EgressProvider {
  attach(ctx: EgressContext): Promise<EgressBinding>;   // once per run/agent, before any outbound call
  detach(ctx: EgressContext): Promise<void>;             // tear down any per-run sidecar
  onDecision?(cb: (d: EgressDecision) => void): void;  // structured audit -> event bus / OTEL
}

This is the only thing core knows. It is wired in exactly like the existing spawnFn dependency-injection seam: pass a real provider in production, a fake in tests, no module mocks.

Smithers core · plumbing

Threading it through both agent paths

Core calls provider.attach() once per run, then propagates the binding to wherever agents actually make requests. Three honest insertion points, all of which exist today.

CLI agents

BaseCliAgent already merges an env overlay into the spawned command env. Add the binding's env to that merge.

const env = {
  ...process.env,
  ...binding.env, // proxy + CA
  ...this.env,
}

SDK agents

The ai SDK accepts a custom fetch. Set a global undici ProxyAgent dispatcher + NODE_EXTRA_CA_CERTS so api.anthropic.com calls route through the proxy.

setGlobalDispatcher(
  new ProxyAgent(binding.env.HTTPS_PROXY))

MCP tools

createMcpToolset spawns servers with a StdioClientTransport that takes env. Merge the binding in so MCP egress is governed too.

new StdioClientTransport({
  env: { ...binding.env, ...cfg.env }
})

Result: one attach() governs CLI subprocesses, SDK fetches, and MCP servers alike. Core never imports a proxy.

Outside core · optional package

The iron-proxy adapter

@smithers-orchestrator/iron-proxy  ·  a thin EgressProvider, importable only by people who want it

import { EgressProvider } from "smithers-orchestrator/egress";

export function ironProxyProvider(opts): EgressProvider {
  return {
    async attach(ctx) {
      const { port, caCertPem } = await startOrReuseIronProxy(opts);  // sidecar or shared listener
      return {
        env: {
          HTTP_PROXY:  `http://127.0.0.1:${port}`,
          HTTPS_PROXY: `http://127.0.0.1:${port}`,
          NO_PROXY:    "127.0.0.1,localhost",
          NODE_EXTRA_CA_CERTS: writeCa(caCertPem),
        },
        caCertPem,
        secretBindings: opts.secrets,   // sk-proxy-XXX -> "anthropic"
      };
    },
    async detach(ctx) { /* stop sidecar if owned */ },
    onDecision(cb) { tailIronProxyAuditJson(cb); },
  };
}

Maybe 150 lines. All vendor specifics live here. Swap in mitmproxyProvider or squidProvider without touching core.

Plue · iron-proxy ON by default

Wiring the default into the cloud

internal/services/agent_dispatch.go · buildServiceSpec()

// 1) Run iron-proxy as a sidecar systemd unit inside the Freestyle microVM.
d.units = append(d.units, ironProxySystemdUnit(d.svc.ironProxyCfg))

// 2) Point the agent at the proxy and trust its CA  (the vendor-neutral contract).
d.agentServiceSpec.Env["HTTP_PROXY"]          = "http://127.0.0.1:8080"
d.agentServiceSpec.Env["HTTPS_PROXY"]         = "http://127.0.0.1:8080"
d.agentServiceSpec.Env["NODE_EXTRA_CA_CERTS"] = "/etc/iron-proxy/ca.crt"

// 3) The sandbox only ever sees a placeholder. iron-proxy holds the real key.
d.agentServiceSpec.Env["ANTHROPIC_API_KEY"] = "sk-proxy-" + d.runID   // was plaintext sk-ant-...

// 4) Belt and suspenders: Freestyle default-deny so the proxy cannot be bypassed.
d.vmReq.Firewall = &freestyle.FirewallPolicy{
  DefaultEgressAction: "deny",
  EgressAllow: []freestyle.FirewallEgressRule{{ Host: "127.0.0.1", Port: 8080 }},
}

Reuses the FirewallPolicy type and the deny-default pattern Plue already runs for workflow sandboxes (buildFirewallPolicy). Agent VMs finally get the same treatment, plus secret rewriting on top.

Plue · deployment choices

Where the proxy sits

TopologyHowBypass resistanceFit
Sidecar in VM iron-proxy as a systemd unit; HTTP_PROXY → localhost Low on its own. Pair with Freestyle deny-default (allow only :8080). Phase 1 default. Simplest.
DNS + nftables VM DNS points at iron-proxy; kernel rules block direct egress High. Hardcoded IPs cannot escape. Phase 2 hardening.
TPROXY Transparent kernel intercept (IP_TRANSPARENT) Highest. Nothing routes around it. Strongest, most setup.
Workspace pods GKE NetworkPolicy → iron-proxy Service / sidecar Medium. Workspaces need broad egress. Audit + secret-rewrite, looser allowlist.

Agent VMs want a tight allowlist. Interactive workspaces want full reach with audit. Same adapter, two policy profiles.

The security upgrade

Secrets the sandbox can never read

Today

VM env: ANTHROPIC_API_KEY = sk-ant-REAL…

Readable via env, ps, a memory dump, or any log line. A prompt-injected step curls it to an attacker. Game over, and the key works anywhere.

With iron-proxy

env: …= sk-proxy-7f3
→iron-proxy swaps for real key→api.anthropic.com only

The sandbox holds a placeholder. The real key lives in the proxy. An exfiltrated sk-proxy-7f3 is worthless off-box, and the swap only happens for the allowlisted host.

This directly closes Plue's plaintext-secret gap. It needs no agent cooperation: the agent keeps sending Authorization: Bearer <placeholder> and never learns the difference.

Observability

Every request, on the record

iron-proxy emits one structured JSON entry per request. The adapter forwards it through onDecision onto the surfaces Smithers and Plue already have:

  • Smithers' event bus as egress.decision events, alongside existing run events.
  • Plue's OTEL export to Cloud Logging, the same path the proxy supports natively (Axiom, Logfire).
  • A per-run egress timeline surfaced in the run UI: who tried to reach what, allowed or denied, and which secret was swapped.

Denials become alerts. A burst of blocked hosts on a run is a strong tamper signal.

One audit record

{
  "host": "api.anthropic.com",
  "method": "POST",
  "action": "allow",
  "status": 200,
  "rule": "allowlist:*.anthropic.com",
  "secretSwapped": "anthropic",
  "runId": "run_8f12",
  "durationMs": 412
}

Beyond the network layer

Two transforms that fit Smithers exactly

MCP policy

iron-proxy can enforce a default-deny tool allowlist for MCP servers: it filters tools/list and rejects denied tools/call.

Smithers spawns MCP servers through createMcpToolset. Routing that traffic through the proxy means tool exposure becomes policy, not trust. A poisoned MCP server cannot offer a tool you never allowed.

Judge

For requests that pass the allowlist but still look risky, the judge transform asks an LLM to allow or defer, with timeouts and a circuit breaker.

Smithers already orchestrates models. A judge call is a natural extension of the same machinery, gating the gray-area egress that a static allowlist cannot classify.

Both are opt-in transforms in the adapter config. Core stays unaware; Plue decides the posture.

Why this shape holds up

The seam earns its keep

  • Core stays clean and testable. A fake provider in tests, a real one in prod. Matches the existing spawnFn DI idiom, no module mocks.
  • Open-source users keep control. Bring your own proxy, run wide-open, or point at a corporate proxy. All speak the same contract.
  • Cloud gets defense in depth by default. Default-deny egress, secret rewriting, audit, with no per-workflow opt-in.
  • It composes. Because the seam is the standard proxy contract, it stacks with enterprise proxies and existing firewall rules.
  • One choke point. Network policy, secret handling, and audit converge in a single place instead of being smeared across every agent.
  • Reuses what exists. Plue's FirewallPolicy, Smithers' env overlay and event bus. Mostly wiring, little new surface.

Honest open questions

What still needs answers

  • Freestyle firewall semantics. Plue's own TODO notes the hosted egress-enforcement contract is not finalized. The sidecar works regardless; nftables is the fallback if the hosted firewall is weak.
  • CA trust distribution. The MITM CA must be present and trusted in every sandbox image, including for non-Node runtimes (Python, Go, curl) that ignore NODE_EXTRA_CA_CERTS.
  • Global dispatcher is process-wide. Setting one undici ProxyAgent affects the whole process. Fine in a per-run VM; needs care if many runs share one Smithers process.
  • Streaming. SSE and websocket model streams must pass through cleanly. iron-proxy proxies both natively; verify under load.
  • TLS-MITM cost. Leaf-cert generation and re-encryption add latency. Measure against agent call volume.
  • Workspaces differ from agents. Interactive workspaces need broad egress. They get audit and secret-rewrite, not a tight allowlist.

Recap

One port, two postures

Smithers core exposes a vendor-neutral EgressProvider seam built on the universal proxy contract, and never names iron-proxy. Plue registers iron-proxy as the default adapter, so every cloud agent gets default-deny egress, boundary secret rewriting, and a full audit trail out of the box.

core: pluggable, no vendor lock-in cloud: secure by default mostly wiring, little new surface
