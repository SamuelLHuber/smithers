# Author the implementation workflow

Author the bespoke implementation workflow at .smithers/workflows/smithering-impl.tsx.

First, ground yourself in the real API — run "bunx smithers-orchestrator docs", read the
relevant fragments, and study the closest seeded examples in .smithers/workflows/
(implement.tsx, mission.tsx, kanban.tsx, research-plan-implement.tsx). Copy structure from
the closest one, then adapt. Do not invent API.

Hard contract for the generated workflow:

- Implements docs/planning/06-orchestration.md exactly (worktrees, merge policy, test
  tiers, model assignment, concurrency, observability, fresh-context worker prompts).
- NEVER pin an agent's `cwd` for a worker that runs inside a `<Worktree>`. The engine
  resolves a worker's working directory as `agent.cwd ?? worktreePath ?? repoRoot`, so a
  pinned `cwd` (e.g. `new ClaudeCodeAgent({ cwd: process.cwd() })` or `new CodexAgent({ cwd:
  process.cwd() })`) OVERRIDES the `<Worktree>` isolation: the worker writes its code into the
  repo root instead of its ticket's worktree branch. The branch then comes up EMPTY, the merge
  lane lands nothing, and every ticket reports "finished" while 0% actually lands on
  integration. Leave agent `cwd` UNSET and let `<Worktree>` own the working dir (see how
  mission.tsx wires worktree workers — it passes no `cwd`). Helper functions that read/write
  the evidence bundle or run git in a worktree must resolve paths against the actual worktree
  path, never `process.cwd()`.
- PARALLEL TRACKS ARE MANDATORY — the build must NOT be a linear ticket-by-ticket chain.
  Compute the ticket dependency DAG from each ticket's dependsOn, group tickets into
  topological waves, and run every wave as a `<Parallel maxConcurrency={N}>` of independent
  tickets (each ticket = its own worker in its own git worktree so concurrent file writes
  never collide). Only a real dependsOn edge may force sequencing; the wall-clock critical
  path should be the DAG depth, not the ticket count.
- Imports tickets from artifacts/smithering/tickets.json at module load; task ids derive
  from ticket ids (stable across resume — never indexes/timestamps).
- Input schema includes a boolean "smoke" (default false): smoke=true processes ONLY the
  first ticket end-to-end (including its verification) with NO approval gates, and must
  reach status finished.
- Wires every blocking gate from docs/planning/04-backpressure.md into pre-merge checks;
  evidence persisted under artifacts/smithering/build/ per ticket id. Evidence paths MUST be
  run-scoped (include `ctx.runId`, e.g. `artifacts/smithering/build/${ctx.runId}/<ticket>/`)
  OR the workflow must delete the whole evidence directory at run start before any done-check
  can read it. A stale `verify.json` from a previous run must never satisfy a fresh run's
  "is this ticket done?" check. Red-before-green for fixes.
- Reviewer model family ≠ implementer model family on every ticket.
- Every loop has maxIterations; side-effecting custom tools declare sideEffect +
  idempotency keys; no caching of side-effect tasks.
- CAPTURE VERDICTS FROM DISK, NOT FROM AGENT RETURNS. The loop-exit / "is this ticket done?"
  check MUST read each ticket's verdict from the on-disk evidence bundle the agents write
  (e.g. `verify.json.pass`, `review.json.approved` under artifacts/smithering/build/<runId>/<id>/),
  NOT from the verify/review agents' structured task-return values. Agent structured returns
  are unreliable and frequently come back empty (they persist their verdict to the bundle and
  emit prose) — gating "done" on them deadlocks the loop: the ticket can NEVER reach done and
  every ticket blocks. Treat the INDEPENDENT test-authority verifier's on-disk `pass` (it
  re-runs every pre-merge gate LIVE and confirms red-before-green) as the AUTHORITATIVE landing
  signal. Surface the cross-family reviewer's findings to the implementer as feedback, but keep
  the review ADVISORY — it must not hard-block a ticket the independent verifier proved green
  over evidence-log hygiene. Safety/security tickets additionally require the adversarial
  challenge verdict to pass.
- `.mdx` prompt files are parsed as MDX modules. When writing prose examples that look like
  JSX or HTML, wrap them in inline backticks or fenced code blocks (for example
  `<Parallel maxConcurrency={N}>`) so the prompt module's default export still compiles.
- NEVER merges to {props.baseBranch}, no exceptions: all work lands on an integration
  branch; merging that branch is a human act after delivery.
- Workers persist HTML decision docs for judgment calls under
  artifacts/smithering/decisions/ (alternatives considered, example inputs/outputs, diffs).
