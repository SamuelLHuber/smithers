## The workflow you ship today is a snapshot

Look at any team running a non-trivial agent in production. You'll see a workflow. Could be DAX's `research → plan → implement`. Could be Gstack's high-token agentic loop. Could be Symfony's straight-line ticket implementer. Could be `parallel-tickets`, which we ship. They're all good. They're all locally optimal for the model and the problem they were tuned for.

They are also all snapshots. Every one of them was tuned weeks or months ago, against a different model, against a different set of MCP servers and CLI tools, with different evaluation criteria. The flow that wins this quarter loses the next. This is not a flaw of the flows. The flow is doing its job. The question is whether *the layer your flow lives on top of* lets you change it without rebuilding the world.

## The wrinkle nobody mentions: agents writing workflows

Here's the part that's underdiscussed.

Workflows aren't only retuned by humans changing their minds. They are increasingly retuned by *the agent itself*. Wrap a good workflow in a self-improving outer loop — the Hermes-shaped thing, where one agent watches another agent's traces and proposes edits to the workflow's source code — and you don't need a human to notice the meta has shifted. The outer loop notices. The outer loop edits. By Thursday next week the workflow your agent is running is one no human author ever wrote.

This is fine — good, even — *if* your workflow is something an agent can read, edit, and re-run safely. It is bad if the workflow is a hand-tuned production cathedral that you can't touch without two days of staging tests.

Whatever you build for the next twelve months, assume the workflow author is sometimes a person and sometimes a model. The author writes, the substrate runs, and both jobs need to stay clean.

## What stays still

Take a long-running agent run — a 45-minute background agent doing a code migration — and write down what it actually has to do at the runtime layer. Not what kind of agent it is. Not which model. Just the *machinery* underneath.

It runs steps. Some of those steps fail and have to retry. It writes intermediate state somewhere durable, because if the host crashes you don't want to throw away 38 minutes of progress. It runs some steps in parallel. It pauses when it needs a human, and wakes back up when the human shows up. It emits events you can read with the tools you already have, so when something explodes at 3am you can figure out which step exploded and why.

That's the boring layer. That layer was true in 2003. It's true now. It will be true in 2030.

The interesting question is whether your infrastructure makes that layer *trivial* to use, or whether it asks you to reinvent it.

## Two developers, overheard

> **Alice:** *We're using LangGraph. It's working fine.*
>
> **Bob:** *What happens when graph-shaped control flow stops being the right answer?*
>
> **Alice:** *Why would it stop being the right answer?*
>
> **Bob:** *Because three months ago role-based crews were the right answer, and four months before that ReAct was the right answer.*
>
> **Alice:** *...*
>
> **Bob:** *I'm just asking — what's the stable part of your stack? The topology, or the runtime underneath?*

Bob's onto something Alice doesn't see yet. There's a difference between two abstractions that often get conflated:

- The **topology** — what shape your agent loop is. Graph, crew, ReAct, planner-executor, swarm, autonomous mission, the next one nobody's named yet.
- The **execution substrate** — durable steps, retries, persistence, parallelism, suspension, resumption, observability.

Frameworks that hardcode a topology age badly. They wager that one shape is the right shape forever, and that wager almost never pays. Frameworks that abstract the *substrate*, on the other hand, age fine — because the substrate in 2027 is going to look basically like the substrate in 2003.

The advice you keep hearing is "don't let a framework hide your prompts and responses." That's good advice for the topology layer. It's bad advice for the substrate underneath. Don't write your retry loop by hand. Don't write your durable state machine by hand. Don't reinvent observability. Use the boring layer that everyone agrees on, and put your taste into the part that actually changes.

## Smithers's bet

Smithers is a bet on the substrate. It does not pick a topology for you. It gives you a primitive — a durable, retryable, observable task — and lets you compose it into whatever topology your problem and your model want this quarter.

Here's the smallest workflow you can write that survives a process crash, in Smithers's [Effect API](/effect/overview):

```ts
import { Smithers } from "smithers-orchestrator";
import { Effect, Schema } from "effect";

const Review = Smithers.createWorkflow({
  name: "review",
  input: Schema.Struct({ repo: Schema.String, sha: Schema.String }),
}).build(($) => {
  const analyze = $.step("analyze", {
    output: Schema.Struct({
      summary: Schema.String,
      risk: Schema.Literal("low", "medium", "high"),
    }),
    timeout: "2m",
    retry: { maxAttempts: 3, backoff: "exponential", initialDelay: "1s" },
    run: ({ input, heartbeat, signal }) =>
      Effect.gen(function* () {
        heartbeat({ phase: "analyzing" });
        return yield* analyzeRepo(input, { signal });
      }),
  });

  const report = $.step("report", {
    needs: { analyze },
    output: Schema.Struct({ markdown: Schema.String }),
    run: ({ analyze }) => ({
      markdown: `# Review\n\n${analyze.summary}`,
    }),
  });

  return $.sequence(analyze, report);
});

await Effect.runPromise(
  Review.execute({ repo: "acme/api", sha: "abc123" }).pipe(
    Effect.provide(Smithers.sqlite({ filename: "smithers.db" })),
  ),
);
```

What you wrote: two steps in a sequence, with retries.

What you got for free:

- **Durable persistence.** Each `$.step` writes its output to SQLite, decoded against the `output` schema. Host dies between `analyze` and `report`? Restart with the same `runId` and `analyze` doesn't re-execute. `report` just resumes.
- **Default-on retries.** LLM APIs fail constantly — 429s, 529s, partial JSON, schema-invalid output, "the model declined to respond." Smithers retries by default, by default, because in 2026 you should not be writing that loop yourself.
- **Cancellation propagation.** `signal` is wired through. Heartbeats publish liveness. The supervisor catches zombies.
- **Effect-native composition.** The whole thing returns an `Effect`. It composes with the rest of your Effect program — services, layers, scopes, fibers — without framework-shaped ceremony.

That last point matters more than it sounds. When your orchestrator is built on `effect`, `Effect.gen`, `Schema`, `Layer`, and `Schedule`, you don't need a parallel set of primitives for "things that are workflow-shaped." A retry policy is just a `Schedule`. A dependency is a `Layer`. An output schema is a `Schema.Struct`. A timeout is `Effect.timeout`. Smithers borrows all of it instead of inventing a parallel ecosystem you'll throw out in a year.

## Background agents are a different animal

There's a shift happening, and most agent stacks are extruded synchronous chat agents that don't survive it.

Synchronous chat is forgiving. The user is staring at the screen. If something fails, the user retries, and an upstream Lambda with a five-minute timeout is fine. Nobody cares about your durable execution story when the entire interaction lives inside a single HTTP request.

Background agents are different. They run for hours. They wake up to events. They survive deploys. They coordinate with other background agents that are also running for hours. They pause for a human approval that won't arrive until tomorrow morning. They wake back up at the right step when the human finally shows up.

You cannot fake this with a queue and a database. You can build it that way, but you'll be reinventing about 60% of what Smithers already does, more poorly, and you'll be doing it instead of building your actual product.

In Smithers it's all one shape. `$.step` is durable. `$.signal` and `$.waitForEvent` durably suspend. `Approval` and `HumanTask` block on a human writing to the database. The supervisor (`smithers supervise`) watches for stale heartbeats and resumes runs that died. A workflow can sleep for a week and wake up on the right step. A workflow can crash mid-task and the next process picks it up. None of this is application code you write.

<Warning>
**Watch it! Sandboxes are not orchestrators.**

A sandbox solves isolation: each task gets its own filesystem, its own kernel namespace, its own port 5173 to fight over. That's a real problem and you should solve it.

But a sandbox doesn't tell you which step is in flight, which step is done, which step crashed, which step is blocked on a human, what the dependency graph is, or how to resume. That's a different problem at a different layer.

The orchestrator should *drive* sandboxes, not live inside one. They're doing different jobs at different layers. Stack them. Don't merge them.
</Warning>

Smithers's sandbox surface — still being finalized; we keep changing our mind on the right abstraction here — lets you run a child workflow, or a single step, in an isolated runtime. Whole-graph sandbox? Fine. Per-step sandbox? Also fine. Mix? Fine. Pluggable provider, you write the strategy.

## A pattern is a composition, not a product

Here's the part that gets people excited about Smithers when it clicks.

The "patterns" you keep reading about — orchestrator-workers, judge-and-jury, planner-executor, fan-out-fan-in, ReAct — are not ground truths. They are *compositions* of a small handful of primitives: a step, a parallel block, a loop, a branch, a signal. Once your substrate hands you those primitives in a sane shape, you can build any of those patterns yourself in an afternoon.

We've watched a lot of teams build a lot of agents, and the same compositions show up over and over. Rather than make you reinvent the producer-and-reviewer pair for the eighteenth time, Smithers ships them as components on top of the primitive — *not in place of it.*

| Component | What it composes |
| :-- | :-- |
| [`ReviewLoop`](/components/review-loop) | Producer + reviewer looping until reviewer approves or `maxIterations` hits. Writer-and-editor, every time. |
| [`Optimizer`](/components/optimizer) | Generator + evaluator looping until a numeric score crosses a threshold. The pattern behind every prompt tuner. |
| [`ScanFixVerify`](/components/scan-fix-verify) | Scanner finds issues, fixers run in parallel, verifier confirms each fix, retries the survivors. Lint-and-fix, codemods, dependency upgrades. |
| [`Panel`](/components/panel) | N specialist reviewers in parallel, moderator synthesizes. Vote, consensus, or synthesize strategies. |
| [`Debate`](/components/debate) | Proposer and opponent argue for N rounds, judge issues a verdict. Adversarial pressure before high-stakes decisions. |
| [`GatherAndSynthesize`](/components/gather-and-synthesize) | Fan out to multiple sources in parallel, fan in through a synthesizer. Every research agent. |
| [`ClassifyAndRoute`](/components/classify-and-route) | Classifier sorts items into categories; each category's specialist handles its items in parallel. Every triage queue. |
| [`EscalationChain`](/components/escalation-chain) | Try tier 1, escalate to tier 2 if confidence is low, escalate to a human if needed. |
| [`Poller`](/components/poller) | Poll an external condition with backoff until satisfied or timed out. Every "is the build green yet." |
| [`Supervisor`](/components/supervisor) | Boss plans, workers execute in parallel, boss reviews and re-delegates failures. |
| [`Saga`](/components/saga) | Forward steps with compensations that run in reverse on failure. For work that *can* be undone and you need to undo it correctly when something blows up halfway through. |

None of these are baked into the runtime. They are pure compositions of `$.step`, `$.parallel`, `$.loop`, `$.match`. You can read the source, fork them, write your own. When the next pattern with no name yet shows up — and it will — you'll compose it from the same primitives, and it'll be durable and observable for free.

## Plot twist: it's also React

Everything above describes Smithers via its [Effect API](/effect/overview), because the Effect framing is the honest one. Durable steps. Retries-as-`Schedule`. Parallelism-as-fibers. Schema-validated outputs. Layered services. If you already think in `Effect.gen`, this is the API.

There is also a [JSX API](/jsx/overview). The exact same workflow, written as React:

```tsx
<Workflow name="review">
  <Sequence>
    <Task id="analyze" output={outputs.analysis} agent={analyst} retries={3}>
      Analyze {ctx.input.repo}@{ctx.input.sha}
    </Task>
    <Task id="report" output={outputs.report}>
      {(ctx) => `# Review\n\n${ctx.output(outputs.analysis).summary}`}
    </Task>
  </Sequence>
</Workflow>
```

Same runtime. Same SQLite. Same retries. Same time travel. Same components. Different authoring surface.

Why both? Because LLMs are unreasonably good at writing React. They have seen more JSX than any other shape of TypeScript by an order of magnitude. When you ask Claude or GPT to author a complex orchestration as JSX with `<Task>`, `<ReviewLoop>`, `<Sandbox>`, `<Approval>`, it produces correct, idiomatic, durable Smithers on the first try at a rate the equivalent Effect-builder authoring just doesn't yet match. Not because the Effect API is worse — because the model's prior is JSX-shaped.

This is a cheat code, and it's the whole reason we made Smithers a JSX runtime in the first place. Remember the Hermes wrinkle from earlier — the workflow you want is a moving target, and the substrate has to be composable enough that *the agent itself* can keep editing the workflow. That works dramatically better when the workflow is JSX, because the editor on the other side of the keyboard is also a model, and the model is fluent in React.

So: Effect API for the people who already think in `Effect.gen`. JSX API for everyone else, and especially for the agents that will be writing the next round of workflows for you. ([Why React?](/why-react) is the long version of this argument.)

## Bullet Points

- **Workflows expire.** The flow you ship today is tuned for today's model and today's tools. Hand-tuned, hard-coded flows can't be re-tuned safely by the outer agent that should be re-tuning them. Composable substrate-based ones can.
- **The substrate is the stable bet.** Durable steps, retries, persistence, parallelism, suspension, resumption, observability. That layer was true in 2003 and it'll be true in 2030. That's where you want your multi-year infra commitment.
- **Topology is fashion. Substrate is plumbing.** Frameworks that hardcode a topology — graphs, crews, swarms — wager that one shape is the right shape forever. The wager almost never pays. Frameworks that abstract the substrate age fine.
- **Background agents are the new shape, and they need durable orchestration.** Synchronous chat is forgiving. Background agents are not. A queue and a database aren't enough.
- **Sandboxes solve isolation, not orchestration.** The orchestrator drives the sandboxes. They're complementary layers. Stacking them is right; merging them is the expensive mistake.
- **Patterns are compositions, not products.** `ReviewLoop`, `Panel`, `Debate`, `ScanFixVerify`, `EscalationChain` — all of these are compositions of `$.step`, `$.parallel`, `$.loop`, `$.match`. Smithers ships them as components, but they live on top of the primitive, not in place of it.
- **Default workflows are examples, not a hidden runtime.** Fork them. They should take half an hour to make your own.
- **JSX is a cheat code for agent authorship.** Effect API for Effect users. JSX API for everyone else, and especially for the agents that will be writing the next round of workflows.
