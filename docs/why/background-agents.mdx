Every six months the right way to build an AI agent changes. So the only sane thing to build is the layer that doesn't.

In 2023 the consensus was RAG: vector DBs everywhere, virtual memory to dodge the 4K window. Then context windows blew up to a million tokens and that whole stack became dead weight. Then Anthropic shipped *Building Effective Agents* and we all dutifully wired up prompt chaining, routing, orchestrator-workers. Context engineering became real work. We needed a browser. No, we needed MCP. We were going to build a hundred specialist sub-agents with cute job titles. No — models got better, generic agents with great prompts won. OpenAI capitulated to MCP. CLIs ate MCPs. We needed sandboxes. We were building software factories. We were synchronizing context across agents. We were running fleets of background agents.

If you coupled your infrastructure to any one of these patterns, you've already rebuilt twice. You will rebuild again.

And background agents — the actual durable, long-running, parallel kind that show up to work without you babysitting them — are here now. Most orchestration stacks were never designed for them, because most orchestration stacks were designed around whatever pattern was fashionable on the week they were written.

This is the post for what to build on instead.

## The hard-coded flow is a snapshot, not a tool

Right now there are three or four genuinely good agent workflows in circulation. DAX's `research → plan → implement`. Gstack's high-token agentic loop. Symfony's straight-line ticket implementer. We ship one called `parallel-tickets` that's pretty good. They are good *today*. They are local maxima frozen at the moment they shipped.

The interesting part isn't picking the right one. It's that the workflow that works for you, on your codebase, with your model, this quarter, is a moving target. And once you wrap your workflow in a self-improving outer agent — Hermes-style — and let *it* edit the workflow, the moving target moves on its own. A frozen, hard-coded flow can't do that. A composable orchestration layer can.

So the question stops being "which flow do I copy?" and becomes "what's the substrate I write flows on top of, that doesn't get thrown away when the meta shifts?"

## The framework trap

The honest critique of agent frameworks is that they are bets on which agent pattern wins. LangGraph encodes graph-based control flow as the paradigm. Crew encodes role-playing crews. Mostly the advice from people who have built one of these in anger is: "start with the raw LLM API, don't let a framework hide your prompts and your responses, don't abstract the topology."

That's good advice for *the prompt-and-response layer*. It's bad advice for the layer underneath.

Because the layer underneath — durability, persistence, retries, parallel execution, event-driven recovery, observability — *doesn't change*. RAG comes and goes. MCP comes and goes. A workflow needing to survive a process crash and resume from the last completed step was true in 2003 when Workflow Foundation shipped and is true now. The treadmill is at the prompt and topology layer. The substrate underneath is boring and stable. The mistake is conflating them.

Smithers is a bet that the boring layer is the one worth shipping a library for. It does not pick a control flow paradigm for you. It gives you a primitive — a durable, retryable, observable task — and lets you assemble whatever topology your model and your problem want this quarter.

## The substrate

Here is the smallest thing you can write in Smithers, in its [Effect API](/effect/overview), that survives a crash:

```ts
import { Smithers } from "smithers-orchestrator";
import { Effect, Schema } from "effect";

const Review = Smithers.createWorkflow({
  name: "review",
  input: Schema.Struct({ repo: Schema.String, sha: Schema.String }),
}).build(($) => {
  const analyze = $.step("analyze", {
    output: Schema.Struct({
      summary: Schema.String,
      risk: Schema.Literal("low", "medium", "high"),
    }),
    timeout: "2m",
    retry: { maxAttempts: 3, backoff: "exponential", initialDelay: "1s" },
    run: ({ input, heartbeat, signal }) =>
      Effect.gen(function* () {
        heartbeat({ phase: "analyzing" });
        return yield* analyzeRepo(input, { signal });
      }),
  });

  const report = $.step("report", {
    needs: { analyze },
    output: Schema.Struct({ markdown: Schema.String }),
    run: ({ analyze }) => ({
      markdown: `# Review\n\n${analyze.summary}\n\nRisk: ${analyze.risk}`,
    }),
  });

  return $.sequence(analyze, report);
});

await Effect.runPromise(
  Review.execute({ repo: "acme/api", sha: "abc123" }).pipe(
    Effect.provide(Smithers.sqlite({ filename: "smithers.db" })),
  ),
);
```

Five things are happening that you didn't have to write:

1. Each `$.step` is durable. Its output is decoded against `output` and persisted to SQLite. If the host dies mid-run, restart with the same `runId` and `analyze` doesn't re-execute — `report` just resumes.
2. `retry: { maxAttempts: 3, backoff: "exponential" }` is the default-on shape. LLM calls fail 5–15% of the time on transient API issues; you should not be writing that retry loop yourself in 2026.
3. `signal` and `heartbeat` are wired through. Cancellation propagates. Long-running steps publish liveness so the supervisor can detect zombies.
4. `$.sequence` and dependencies (`needs`) describe the graph. Smithers reads it and schedules.
5. The whole thing is an `Effect`. It composes with the rest of your Effect program — services, layers, scopes, fibers — without ceremony.

That last point is the one most "background agents are here" articles miss: when your orchestrator is built on `effect`, `Effect.gen`, `Schema`, `Layer`, and `Schedule`, you don't need a parallel set of primitives for "things that are workflow-shaped." A retry policy is a `Schedule`. A dependency is a `Layer`. An output schema is a `Schema.Struct`. A timeout is `Effect.timeout`. The orchestrator borrows all of it, instead of inventing a parallel ecosystem you'll throw away in 18 months.

## The four things that don't change

If you read the *Building Effective Agents* taxonomy and squint, every block ends up needing the same four properties:

**Durable steps.** A completed task is never re-executed. This is non-negotiable for anything that calls `git push`, `stripe.charges.create`, or burns 80,000 tokens per call. In Smithers a `$.step` is durable by construction — its row in SQLite is the contract. There is no "did this finish" question to ask.

**Persistent external state.** Agents need to read what previous agents wrote. Smithers makes the easy path obvious: every output schema becomes a typed SQLite table. You can plug in another store, but the pit-of-success is "your state is already persisted, you didn't have to do anything." Want to inspect it? It's just SQLite. Open it in any tool you already have.

**Parallel work.** `$.parallel(...nodes, { maxConcurrency: 4 })` runs independent steps concurrently and writes their outputs in any order. The runtime is built on Effect fibers, so structured concurrency, interruption, and resource lifetimes all behave the way you expect. You don't have to think about Node's process model — you describe what's parallel and the runtime does the right thing.

**Event-driven control flow.** Smithers's loop is `event in → re-evaluate plan → schedule tasks → events out`. A new event (a step finished, a human approved, an external signal arrived) re-renders the plan; the plan diff is what gets scheduled next. It is a one-way data flow. You don't manage state machines by hand. The plan is a function of the current state, and the state is a function of past events.

Four properties. Stable. The pattern on top can change every six months. These don't.

## Retries and observability are first-class, not bolt-on

LLM APIs fail. Constantly. 429, 529, request timeout, partial JSON, schema-invalid output, sub-quota tokens, "the model declined to respond." Every single one of these is a retry. Smithers retries by default for `agent`-typed steps because you should not have to remember to. The default policy is exponential backoff with jitter, capped attempts, schema-aware error classification.

Observability is the same — you should not have to bolt it on after the fact. Smithers writes structured events to SQLite (every state transition, every attempt, every input/output, every retry, every approval) and exposes Prometheus metrics out of the box. We deliberately did not invent a bespoke trace format; your existing tooling already speaks Prometheus and SQL. The CLI gives you `smithers timeline <runId>`, `smithers logs <runId> --follow`, `smithers diff <runId:frame1> <runId:frame2>`, and time-travel via `smithers fork --frame N --reset-node analyze`. None of this is opt-in.

When something fails at 3am and you wake up to 14 stuck runs, the value of "every state transition is a SQL row" cannot be overstated.

## Composable patterns, not blessed architectures

There is a real critique of frameworks that pick a topology — graphs, crews, swarms — and then everything you build has to fit. The trap is the topology, not having opinions. Smithers's opinion is that *topologies are libraries*, not core. We've watched enough teams build enough agents to notice the patterns that show up over and over, and we ship them as components on top of the primitive — not in place of it.

The current ones, none of which are baked into the runtime:

- [`ReviewLoop`](/components/review-loop) — a producer agent and a reviewer agent loop until the reviewer says approved or `maxIterations` is hit. The pattern that shows up every time someone builds "writer + editor."
- [`Optimizer`](/components/optimizer) — generator and evaluator loop until a numeric score crosses a threshold. The pattern behind every prompt tuner, every RLAIF setup, every "make this better until it's good enough."
- [`ScanFixVerify`](/components/scan-fix-verify) — scanner finds issues, fixers run in parallel, verifier confirms each fix, retries the survivors. The pattern behind every lint-and-fix, every codemod, every dependency upgrade.
- [`Panel`](/components/panel) — N specialist reviewers in parallel, moderator synthesizes. Vote, consensus, or synthesize strategies.
- [`Debate`](/components/debate) — proposer and opponent argue for N rounds, judge issues a verdict. Useful when you need adversarial pressure before a high-stakes decision.
- [`GatherAndSynthesize`](/components/gather-and-synthesize) — fan out to multiple sources in parallel, fan in through a synthesizer. The pattern behind every research agent.
- [`ClassifyAndRoute`](/components/classify-and-route) — classifier sorts items into categories, each category's specialist handles the items in parallel. The pattern behind every triage queue.
- [`EscalationChain`](/components/escalation-chain) — try tier 1, escalate to tier 2 if confidence is low, escalate to a human if needed.
- [`Poller`](/components/poller) — poll an external condition with backoff until satisfied or timed out. The pattern behind every "is the build green yet."
- [`Supervisor`](/components/supervisor) — boss plans, workers execute in parallel, boss reviews and re-delegates failures. The pattern behind every "manager and team."
- [`Saga`](/components/saga) — forward steps with compensations that run in reverse on failure. For the work that *can* be undone and you need to undo it correctly when something blows up halfway through.

You don't have to use any of them. The point is you can compose your own out of `$.step`, `$.parallel`, `$.loop`, `$.match` in an afternoon, and they'll be durable and observable for free. Or you can mount one of these and stop reinventing the producer/reviewer pair for the eighteenth time.

## Async is where everything breaks

Synchronous chat agents — the user typed something, the agent responds within a turn — are a forgiving shape. The user is staring at the screen. If something fails, they retry. The infrastructure can be a Lambda; nobody cares.

Asynchronous background agents are a totally different problem. They run for hours. They wake up to events. They need to survive a deploy. They need to coordinate with other background agents that are also running for hours. They need to pause for a human approval that won't come until tomorrow. They need to wake back up at the right step when the human finally approves.

This is where most agent stacks fall apart, because most agent stacks are extruded synchronous chat agents. Durable orchestration is non-negotiable for the async case. You cannot fake it with a queue and a database — you can build it that way, but you'll be reinventing 60% of what Smithers already does, badly.

Smithers handles this with one shape: `$.step` is durable, [`Signal`](/components/signal) and [`WaitForEvent`](/components/wait-for-event) durably suspend, [`Approval`](/components/approval) and [`HumanTask`](/components/human-task) block on a human writing to the database, and the supervisor (`smithers supervise`) watches for stale heartbeats and resumes. A run can sleep for a week and wake up on the right step. A run can crash mid-task and the next process picks it up. None of this is application code you write.

## Sandboxes: orchestrate them, don't *be* one

Sandboxes (Codeplane, Modal, Daytona, e2b, bare Docker) are great. They solve isolation: each task gets its own filesystem, its own port, its own kernel namespace, and you stop having parallel agents fight over port 5173.

What sandboxes don't solve is "which step is done, which is in flight, which crashed, which one's blocked on a human, what's the dependency graph, how do I resume." That's a different problem. The two layers are complementary.

Smithers's stance is that the orchestrator should *drive* sandboxes, not live inside one. We expose a [`Sandbox`](/components/sandbox) component (and a builder analog) that lets you run a child workflow — or a single step — in an isolated runtime. You can run the whole graph in one sandbox. You can run each step in its own sandbox. You can mix. The sandbox provider is pluggable; you write the strategy.

This is honestly the part of Smithers where we keep changing our mind on the right abstraction — running cleanly local *and* cleanly in the cloud, generic over providers, with per-task isolation when you want it and shared sandboxes when you don't, is genuinely hard. Expect this surface to lock down soon.

## State-of-the-art workflows, batteries included

The biggest legitimate complaint about Smithers is that "give me primitives" is a worse first impression than "here's a workflow, run it." We hear this. So `smithers init` ships a set of opinionated workflows that we keep current with whatever the state of the art is this quarter:

- [**Kanban-board**](/workflows/kanban) — items flow through ordered columns, each column has an agent. The mental model that just works for support, triage, content production.
- [**Mission-based**](/workflows/mission) — the loop that's been winning lately for autonomous agents on a goal.
- [**Research → Plan → Implement**](/workflows/research-plan-implement) — DAX's flow. Still excellent for code tasks.
- [**Plan + PRD**](/workflows/write-a-prd) — the slow, careful one for high-stakes changes. Pairs with an approval gate.
- [**Grill-me**](/workflows/grill-me) — Matt Pocock's "interrogate the human until you have enough context to write the spec."

Use them as-is. Or — and this is the actual point — fork them. They're written in the same Smithers primitives. You can read them, modify them, and your fork is just as durable, just as observable, just as composable. The default workflows are *examples* of how to use the substrate, not a hidden runtime that lives somewhere you can't touch.

## Smithers Hub

Self-hosted observability, history, time-travel, signal delivery, approval inbox — all of this works locally with `smithers.db`. But running a fleet of background agents in production usually means you want this *somewhere a team can see it* without standing up Prometheus and Grafana yourself.

Smithers Hub is the managed version. Point your runtime at it, you get the dashboard, the history, the approval inbox, the metrics, the alerts, without setting any of it up. Same SQLite-shaped data model underneath; same primitives; same workflows. You can leave at any time and run it yourself.

## The reveal: it's also React

Everything above describes Smithers via its [Effect API](/effect/overview), because the Effect framing is the honest one. Durable steps, retries-as-`Schedule`, parallelism-as-fibers, schema-validated outputs, layered services. If you want to write workflows that compose with the rest of an Effect program, this is the API.

There is also a [JSX API](/jsx/overview). The same workflow as JSX:

```tsx
<Workflow name="review">
  <Sequence>
    <Task id="analyze" output={outputs.analysis} agent={analyst} retries={3}>
      Analyze {ctx.input.repo}@{ctx.input.sha}
    </Task>
    <Task id="report" output={outputs.report}>
      {(ctx) => `# Review\n\n${ctx.output(outputs.analysis).summary}`}
    </Task>
  </Sequence>
</Workflow>
```

Same runtime. Same SQLite. Same retries. Same time travel. Same components. Different authoring surface.

Why both? Because LLMs are unreasonably good at writing React. They have seen more JSX than any other shape of TypeScript by an order of magnitude. When you ask Claude or GPT to author a complex orchestration as JSX with `<Task>`, `<ReviewLoop>`, `<Sandbox>`, `<Approval>` components, it produces correct, idiomatic, durable Smithers on the first try at a rate that the equivalent Effect-builder authoring just doesn't match — yet. Not because the Effect API is worse; because the model's prior is JSX-shaped.

This is a cheat code. The whole pitch of Smithers is that the workflow you want is a moving target, and the substrate has to be composable enough that *the agent itself* can keep editing the workflow. That works dramatically better when the workflow is JSX — because the editor on the other side of the keyboard is also a model, and the model is fluent in React.

So: Effect API for the people who already think in `Effect.gen`. JSX API for everyone else, and especially for the agents that will be writing the next round of workflows for you. ([Why React?](/why-react) goes deep on the JSX side of this argument.)

The treadmill won't stop. Background agents are here, and the next pivot is already being prototyped somewhere on a Discord server. The only sane move is to build on the layer that doesn't change. That's the substrate Smithers exists to be.
