Smithers is a durable runtime for long-running AI coding agents. You drive it through the agent you already use (Claude Code, Codex, Cursor, and more): ask for an outcome, and it runs for minutes or days with crash recovery, retries, human approvals, replay, and full observability.

## Quickstart

One command sets everything up. From inside your project:

```bash
bunx smithers-orchestrator init
```

`init` installs the `smithers` skill into your coding agents (no `mkdir`, no `curl`) and scaffolds a `.smithers/` folder of ready-made workflows. Then just ask:

> **You:** Add rate limiting and keep iterating until the tests pass.

Your agent reaches for the right workflow, starts a durable run, and keeps going through retries and review loops until the work is actually done. A crash mid-run resumes from the last completed step.

## Built-in workflows

`bunx smithers-orchestrator init` scaffolds a `.smithers/` folder preloaded with
production-ready workflows. Ask your agent for the outcome and it runs
`bunx smithers-orchestrator workflow run WORKFLOW_ID --prompt "..."`:

```bash
# break a request into tickets, then implement them on a board
bunx smithers-orchestrator workflow run tickets-create --prompt "add rate limiting and audit logging"
bunx smithers-orchestrator workflow run kanban
```

## How it fits together

<Frame caption="Models make stateless calls, harnesses manage the context window around them, topologies wire agents into patterns, and Smithers runs the whole stack as durable workflows. Everything above the orchestration layer churns; that bottom layer holds still.">
  <img src="/images/why/agent-stack.svg" alt="Four stacked layers: a model layer of stateless calls, a harness layer that manages the context window, a topology layer of orchestration patterns, and a stable Smithers orchestration layer underneath" />
</Frame>

The **model layer** is stateless: a request goes in, tokens come out. The **harness layer** wraps a model and manages its context window (Claude Code, Codex, Pi, opencode). The **topology layer** is the pattern you arrange agents in (ReAct, crew, swarm, plan-execute, subagent fan-out). The **orchestration layer** at the bottom is Smithers: it wires humans, plain code, and harnesses into larger workflows and makes them durable. The layers above it change constantly. This one holds still.

## Author your own (optional)

You never write this by hand to use Smithers; your agent does. But when you want to, a workflow is a small TypeScript tree:

```tsx
/** @jsxImportSource smithers-orchestrator */
import { createSmithers, Sequence, Task } from "smithers-orchestrator";
import { z } from "zod";

const { Workflow, smithers, outputs } = createSmithers({
  hello: z.object({ message: z.string() }),
});

export default smithers((ctx) => (
  <Workflow name="hello">
    <Sequence>
      <Task id="greet" output={outputs.hello}>
        {{ message: `Hello, ${ctx.input.name}` }}
      </Task>
    </Sequence>
  </Workflow>
));
```

Outputs are validated by Zod, persisted to SQLite, and survive crashes. See the
[JSX surface](/jsx/overview) and [Components](/components/workflow) for the full API.

## Any agent, any model

Smithers doesn't bet on one lab or one harness. Point each task at whichever agent is best
for the job, and switch freely:

- **CLI agents**: [Claude Code](/integrations/cli-agents), Codex,
  [Pi](/integrations/pi-integration), Antigravity, Kimi, and more, driven through their
  own runtimes.
- **SDK agents**: any model the [AI SDK](/integrations/sdk-agents) supports, with tools,
  structured output, and MCP.
- **Mix them in one workflow**: a frontier model plans, a fast model fans out, a
  specialized harness does the edits. The workflow doesn't change when the model does.

## Durable and observable

Runs survive crashes, restarts, and flaky tools. Every completed step is persisted the
moment it finishes, so the runtime always knows what's done and what to run next, and you
can rewind, fork, and replay any run.

```bash
bunx smithers-orchestrator up workflow.tsx --run-id abc123 --resume true   # resume after a crash
bunx smithers-orchestrator rewind abc123 --frame 4                         # time-travel to an earlier frame
bunx smithers-orchestrator observability --detach                          # Grafana + Prometheus + Tempo + OTLP
```

Every run emits Prometheus metrics and OpenTelemetry traces, so you can watch token spend,
task latency, retries, and failures across thousands of runs. See
[How It Works](/how-it-works) for the render, execute, persist loop.

## Eval-driven and self-improving

A workflow you cannot measure is a workflow you cannot trust. Write an eval suite of input
cases and expected results, and Smithers runs your workflow across every case and writes a
regression report:

```bash
bunx smithers-orchestrator eval workflow.tsx --cases evals/smoke.jsonl --suite smoke
bunx smithers-orchestrator optimize workflow.tsx --cases evals/smoke.jsonl --suite smoke-gepa
```

`optimize` turns a workflow on itself: it runs your eval suite, uses GEPA to rewrite the
prompts on your agent tasks, re-runs the suite, and keeps the new prompts only when they
score higher. See [Eval Suites](/guides/evals-quickstart) and
[Workflow Optimization](/guides/workflow-optimization).

## Scale across machines

Most workflows run fine on your laptop. When you need isolation, parallelism, or
horizontal scale, the same [`<Sandbox>`](/components/sandbox) primitive runs a child
workflow through an injectable provider (local or remote) with no change to the
workflow:

- **Local**: run agents in an isolated sandbox on your own machine.
- **Remote**: gVisor, Kubernetes, [freestyle.sh](/examples/freestyle-sandbox-provider),
  Daytona, and Cloudflare.
