Smithers is driven by the coding agent you already use (Claude Code, Codex, Cursor, and more). You don't write workflows by hand or memorize CLI verbs. You run one command, then ask for an outcome.

<Steps>

<Step title="Run one command">

From inside your project:

```bash
bunx smithers-orchestrator init
```

That's it. `init` does everything:

- **Installs the `smithers` skill** into the coding agents on your machine, so your agent knows how and when to use Smithers. No `mkdir`, no `curl`.
- **Scaffolds `.smithers/`** with ready-made workflows (`hello`, `implement`, `plan`, `review`, `debug`, and more) your agent can pick from.

<Note>
Already past the basics? `init` also runs `bun install` in `.smithers/` and detects your agents. Skip the skill install with `--no-skill`.
</Note>

</Step>

<Step title="Ask your agent for an outcome">

Now just talk to your agent. It reaches for the right workflow and starts a durable run.

> **You:** Add rate limiting and keep iterating until the tests pass.

The agent picks the `implement` workflow, loops it until your test command is green, and survives a crash mid-run: every finished step is persisted, so a restart resumes from where it stopped.

> **You:** Give me an ordered plan for account-level billing limits. Don't touch the code.

It picks the `plan` workflow: research and a prioritized plan, no edits to your tree.

</Step>

</Steps>

## Make it yours (optional)

Want to watch one run end to end? The seeded `hello` workflow is the smallest example:

```bash
bunx smithers-orchestrator workflow run hello
```

Its entire prompt lives in `.smithers/prompts/hello.mdx`, plain Markdown. Edit it, run it again, watch the output change. That's the on-ramp: tweak a prompt, then copy `.smithers/workflows/hello.tsx` when you're ready to author your own.
